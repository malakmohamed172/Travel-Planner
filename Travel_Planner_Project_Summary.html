1. المشروع بيتكلم عن إيه؟

Travel Planner هو Web Application لإدارة الرحلات الجماعية. الفكرة إن المستخدم يقدر يشوف رحلات، يحجز رحلة، يدفع، يتابع الأنشطة اليومية، يضيف مصاريف، يرفع مستندات، ويستقبل Notifications. أما الـ Leader فيقدر ينشئ رحلات ويقسمها لأيام ومحطات، ويدير الأنشطة المقترحة. والـ Admin عنده Dashboard لإدارة المستخدمين والحجوزات والتقارير.

للمستخدم العادي Member

يشوف الرحلات، يحجز، يدفع، يرفع مستند، يتابع Activities، ويختار Going / Not Going.

للقائد Leader

ينشئ Trip، يضيف Itinerary وStops، ويمتلك صلاحية Approve / Reject للـ Activities الخاصة برحلته.

للأدمن Admin

يشوف Dashboard فيها users/bookings/statistics، ويقدر يعدل أو يحذف مستخدمين.

قاعدة البيانات

بتخزن users, roles, trips, bookings, payments, activities, expenses, documents, notifications.

3. الـ Workflow الأساسي

  1. المستخدم يعمل Register أو Sign in.
  2. السيستم يحفظ بياناته في Session بعد تسجيل الدخول.
  3. لو Admin يروح Admin Dashboard، ولو User يروح Homepage.
  4. الـ Leader ينشئ Trip ويضيف itinerary days وstops.
  5. الـ Member يشوف الرحلات ويحجز Trip.
  6. بعد الحجز، المستخدم يقدر يدفع وتتحول حالة الحجز إلى confirmed.
  7. المستخدمين يقترحوا Activities على itinerary معين.
  8. Leader الرحلة يوافق أو يرفض activity.
  9. Notifications بتوصل لصاحب العلاقة، زي activity proposed أو approved/rejected.

6. شرح التستات

التستات مكتوبة بـ PHPUnit. كل Test Class بيجهز بيانات في setUp، ينظفها في tearDown، وبعدين كل function تبدأ بـ test... تختبر سيناريو معين باستخدام Assertions.

bootstrap.php

ده الملف اللي PHPUnit بيشغله قبل التستات. بيعمل include للـ Controllers وModels المطلوبة، وبيوفر helper اسمها غالبا getTestConnection() عشان كل test يفتح اتصال بقاعدة البيانات. 

protected function setUp(): void
{
    $this->conn = getTestConnection();
    // create test data
}

protected function tearDown(): void
{
    // delete test data
    $this->conn->close();
}

7. أساسيات PHP المستخدمة في المشروع

PHP داخل HTML

PHP يقدر يكتب HTML ديناميكي. أي كود بين <?php ... ?> بيتنفذ على السيرفر قبل ما الصفحة توصل للمتصفح. 

<h1>Welcome, <?= htmlspecialchars($user['name']) ?></h1>

GET

GET بتجيب بيانات من الرابط. مثال: edit.php?id=5 هنا نقدر نقرأ id بـ $_GET['id']. 

$id = isset($_GET['id']) ? (int)$_GET['id'] : 0;

GET مناسب للقراءة أو التنقل، لكن مش أفضل اختيار للحذف أو العمليات الحساسة لأنه ظاهر في URL.

POST

POST بتستخدم غالبا مع forms لما نرسل بيانات زي login/register/create trip. البيانات بتكون في $_POST. 

if ($_SERVER["REQUEST_METHOD"] === "POST") {
    $email = $_POST['email'];
    $password = $_POST['password'];
}

Session

Session بتخزن بيانات المستخدم بين الصفحات. بعد login بنحفظ user id وrole في $_SESSION. 

session_start();
$_SESSION['user'] = [
    "id" => $row['user_id'],
    "email" => $row['email'],
    "role_id" => $row['role_id']
];

require_once

بتستخدم لاستدعاء ملف PHP مرة واحدة فقط. مثلا Controller يحتاج Model أو DBController. 

require_once '../../Controllers/DBController.php';
require_once '../../Models/users.php';

mysqli وPrepared Statements

mysqli هي طريقة PHP للتعامل مع MySQL. Prepared Statement بتحمي من SQL Injection لأنها تفصل كود SQL عن البيانات القادمة من المستخدم. 

$stmt = $conn->prepare("SELECT * FROM users WHERE email = ? LIMIT 1");
$stmt->bind_param("s", $email);
$stmt->execute();
$result = $stmt->get_result();

bind_param types

الحرفالمعنىمثال
sstringemail, name, status
iintegeruser_id, trip_id
ddouble/floatbudget, cost, amount
bblobملفات binary نادرة الاستخدام هنا

password_hash وpassword_verify

ممنوع تخزين الباسورد plain text. password_hash بيحوله لقيمة آمنة، وpassword_verify بتتأكد إن الباسورد اللي المستخدم كتبه يطابق الهاش. 

$hash = password_hash($password, PASSWORD_DEFAULT);

if (password_verify($password, $row['password'])) {
    // login success
}

Redirect بالـ header

بعد عملية ناجحة بننقل المستخدم لصفحة تانية باستخدام header. لازم بعدها نعمل exit عشان الكود اللي بعده مايكملش. 

header("Location: ../Views/User/homepage.php");
exit();

htmlspecialchars

لما نطبع بيانات جاية من المستخدم في HTML لازم نستخدم htmlspecialchars عشان نقلل خطر XSS. 

<td><?= htmlspecialchars($row['name']) ?></td>

8. نقاط أمان وفاليديشن مهمة

  • التحقق من login قبل أي صفحة حساسة باستخدام Session.
  • التحقق من role_id قبل عمليات Admin أو Leader.
  • التحقق من ownership: الـ leader يعدل أو يحذف رحلاته فقط.
  • منع الباسورد الأقل من 8 حروف من السيرفر وليس من الواجهة فقط.
  • استخدام Prepared Statements بدل دمج قيم المستخدم داخل SQL مباشرة.
  • استخدام htmlspecialchars عند عرض بيانات المستخدمين.
  • عدم الاعتماد على إخفاء الزر فقط، لأن المستخدم ممكن يكتب URL بنفسه.

9. ملخص للمناقشة

لو اتسألت المشروع بيعمل إيه: قل إنه Travel Planner لإدارة الرحلات الجماعية، فيه roles مختلفة، وفيه flows للحجز والدفع والأنشطة والمصاريف والإشعارات. استخدمنا PHP مع MySQL وMVC بسيط، والتستات بـ PHPUnit بتغطي أهم العمليات زي auth/trips/bookings/payments/activities/expenses.

ولو اتسألت عن أهم جزء تقني: ركز على Session للـ authentication، role_id للـ authorization، Prepared Statements لحماية SQL، وpassword_hash لحماية الباسورد.
